## 一、产品概述

### 1.1 产品定位

findRightGuy 是一个运行在 Claude Code 上的独立 Skill，通过分析用户与前任的真实聊天记录，以用户自身情绪反应为客观信号，识别触发用户正向情绪的对话特征，跨前任聚合提炼「理想对象画像」，最终服务于用户的自我认知与未来择偶决策。

### 1.2 核心价值主张

> 它不分析前任，它分析你。

用户在关系中的真实情绪反应（emoji、语气词、回复速度、回复长度）是最客观的偏好数据。findRightGuy 将这些分散在聊天记录中的隐性数据结构化，帮助用户看清自己真正喜欢什么样的互动方式和性格特质。

### 1.3 目标用户

| 用户类型 | 描述 | 核心需求 |
|---------|------|---------|
| 主力用户 | 有过 1+ 段感情经历、思考过「我到底喜欢什么样的人」的用户 | 自我认知、择偶参考 |
| 潜在用户 | 正在恋爱中、想了解自己情感模式的用户 | 关系洞察、顾问建议 |
| 技术用户 | Claude Code 用户、对 AI Skill 感兴趣的开发者 | 工具体验、二次开发 |

## 二、背景与问题定义

### 2.1 用户痛点

1. **经验无法沉淀**：每段感情结束后，用户很难系统性地总结「哪些互动让我快乐」，经验随时间流逝
2. **自我认知模糊**：用户通常只能描述「感觉对了」，无法具体说明什么行为、什么风格让自己投入
3. **择偶标准不清晰**：相亲或认识新对象时，缺乏有数据支撑的「择偶参考标准」
4. **聊天记录价值未被挖掘**：微信、QQ 中存储了大量真实的情感数据，但几乎没有工具帮助用户从中提取有用信息

### 2.2 现有方案的局限

| 现有方案 | 局限性 |
|---------|--------|
| MBTI / 星座分析 | 基于通用人格模型，不反映个体真实偏好 |
| 心理咨询 | 成本高、依赖主观回忆，有记忆偏差 |
| 手动总结 | 费时费力，难以量化，容易美化过去 |
| 通用 AI 恋爱建议 | 无个人历史数据，给出的是通用建议 |

## 三、产品目标

### 3.1 核心目标

- 用户通过导入 1 位及以上前任的聊天记录，获得有数据支撑的「理想对象画像」
- 用户能够通过对话体验「理想对象」的互动风格，满足情感模拟需求
- 用户能够导出结构化的「择偶参考模版」，辅助现实中的感情决策

### 3.2 成功指标

| 指标 | 定义 |
|------|------|
| 分析完成率 | 用户成功完成 `/analyze-ex` 全流程的比例 |
| 优质对话识别率 | 在总对话片段中识别出优质片段的比例（基准值：30%+） |
| 化学反应评分覆盖率 | 成功生成 chemistry_score 的前任数据占比 |
| 模版导出次数 | 用户主动触发 `/export-template` 的频率 |
| 画像进化次数 | 用户通过 `/ideal-chat` 学习触发 Correction 写入的次数 |

## 四、功能需求

### 4.1 核心功能模块

#### F1：前任数据分析（/analyze-ex）

**功能描述**：引导用户导入前任聊天记录，完成情绪评分、特征提取和画像更新。

**输入**：
- 前任代号（slug）
- 聊天记录文件（微信 txt/json/csv/html，QQ txt/mht）或口述描述

**处理流程**：
```
信息录入 → 聊天记录解析 → 情绪评分 → 特征提取 → 跨前任聚合 → 版本备份
```

**输出**：
- `data/exes/{slug}/raw_analysis.json`（情绪评分结果）
- `data/exes/{slug}/features.json`（前任特征数据）
- `data/ideal/persona.md`（更新后的理想对象画像）
- 分析摘要（优质对话占比、化学反应评分、新增特征列表）

**优先级**：P0（核心路径）

---

#### F2：理想对象对话（/ideal-chat）

**功能描述**：以理想对象画像为基础，提供持续进化的 AI 对话体验。

**运行规则**：
- 加载 `data/ideal/persona.md` Layer 0-6
- Layer 0 硬规则优先级最高，不可违背
- 实时感知用户情绪信号，动态调整互动风格
- 对话结束后自动提取学习内容写入 Layer 6

**学习机制**：
- 检测到正向信号 → 记录当前行为到候选 Correction
- 用户主动确认「这句话很好」→ 立即写入 Layer 6
- 累计 10 条 Correction → 提示用户重新聚合

**安全边界**：
- 不扮演特定真实前任
- 不强化「我们在一起」等现实混淆表达
- 发现用户情绪明显低落时，优先关心

**优先级**：P0

---

#### F3：恋爱模版导出（/export-template）

**功能描述**：生成结构化的七章节择偶参考模版。

**模版结构**：
1. 他是什么样的人（性格综述）
2. 他怎么跟我说话（风格 + 示例对话）
3. 让我最开心的行为 TOP10
4. 我的自我洞察
5. 红绿灯清单（🟢绿灯 / 🟡黄灯 / 🔴红灯）
6. 各前任贡献图谱
7. 给未来自己的话

**输出**：`data/ideal/template.md` + 版本存档

**优先级**：P1

---

#### F4：恋爱顾问（/advisor）

**功能描述**：基于个人历史数据提供个性化恋爱建议，支持两种场景。

**场景 A：实时对话分析**
- 用户粘贴与新对象的聊天记录
- AI 对照绿灯/黄灯/红灯特征进行扫描
- 输出分类报告 + 综合建议

**场景 B：主动咨询**
- 用户描述恋爱困惑
- AI 基于「我的自我洞察」数据给出个性化回答
- 不替用户做决定，引导用户自我判断

**顾问边界**：
- 不提供「如何挽回前任」建议
- 发现自我否定迹象时，先关心后分析

**优先级**：P1

## 五、情绪评分算法规范

### 5.1 信号权重

| 信号类型 | 权重 | 说明 |
|---------|------|------|
| 正向 emoji（😂🥰😊）| +3.0 | 强正向情绪信号 |
| 正向语气词（哈哈/好呀）| +1.5 | 中等正向信号 |
| 感叹号 | +1.0 | 弱正向信号 |
| 主动追问 | +1.2 | 参与度高 |
| 回复变长 | +2.0 | 投入度上升 |
| 回复变快 | +1.8 | 兴趣度上升 |
| 主动发起新话题 | +2.5 | 强参与信号 |
| 负向 emoji（😒🙄）| -3.0 | 强负向信号 |
| 敷衍回复（哦/嗯）| -2.0 | 中等负向信号 |
| 回复变短 | -1.5 | 投入度下降 |

### 5.2 优质对话判定（四条件同时满足）

1. 片段用户消息数 ≥ 20 条
2. 用户情绪整体趋势为**上升**（线性斜率 > 0.15）
3. 片段内用户情绪**平均分 ≥ 6.0**
4. 情绪**峰值分 ≥ 7.5**

### 5.3 跨前任加权融合

```
weight = chemistry_score × quality_ratio
merged_score = Σ(score_i × weight_i) / Σ(weight_i)
```

冲突检测：同维度差异 > 3 分时，标注 `⚠️ 偏好未定`，展示两种风格样本供用户选择。

## 六、数据模型

### 6.1 文件结构

```
data/
├── exes/
│   └── {slug}/
│       ├── parsed.json         # 聊天记录解析结果
│       ├── raw_analysis.json   # 情绪评分结果
│       ├── features.json       # 前任特征数据
│       └── memories/           # 原始素材
├── ideal/
│   ├── persona.md              # 理想对象画像（Layer 0-6）
│   ├── template.md             # 最新恋爱模版
│   ├── aggregated.json         # 聚合数据
│   ├── SKILL.md                # 可运行 Skill 文件
│   └── versions/               # 历史版本快照
└── meta.json                   # 全局元数据
```

### 6.2 理想对象画像分层结构

| Layer | 名称 | 内容 | 可修改性 |
|-------|------|------|---------|
| Layer 0 | 硬规则 | 不可违背的行为边界 | 不可修改 |
| Layer 1 | 身份锚定 | 融合来源、主要性格维度 | 自动生成 |
| Layer 2 | 说话风格 | 口头禅/emoji/幽默感/话题引导 | 自动 + 可纠正 |
| Layer 3 | 情感模式 | 开心触发器/最佳互动类型 | 自动 + 可纠正 |
| Layer 4 | 关系行为 | 联系频率/边界感/不喜欢的行为 | 自动 + 可纠正 |
| Layer 5 | 冲突提示 | 跨前任偏好冲突标注 | 用户决策 |
| Layer 6 | Correction | 对话学习动态追加 | 对话自动写入 |

## 七、非功能需求

### 7.1 数据安全

- 所有数据仅存储在本地 `data/` 目录
- 不上传任何服务器，不调用外部 API 处理用户数据
- `data/` 目录已加入 `.gitignore`，不随代码提交

### 7.2 版本管理

- 每次 `/analyze-ex` 完成后自动备份
- 保留最近 10 个版本，更早的自动清理
- 支持 `/rollback` 回滚到任意历史版本

### 7.3 容错降级

- Bash 工具执行失败时，提示用户手动执行命令
- AI 可直接读取 JSON 文件，用 Write/Edit 工具降级生成目标文件

### 7.4 运行环境

| 依赖 | 版本要求 |
|------|---------|
| Claude Code | 最新版 |
| Python | 3.9+ |
| Pillow | ≥ 9.0.0 |
| chardet | ≥ 4.0.0 |
| python-dateutil | ≥ 2.8.0 |

## 十、约束与边界

1. 本项目为**完全独立 Skill**，不依赖任何外部 Skill 子仓库
2. 不在任何文件中暴露前任真实姓名（对外使用代称）
3. 不提供「如何挽回前任」功能
4. 不鼓励用户沉溺于 AI 模拟关系，发现异常依恋迹象时温和提示
